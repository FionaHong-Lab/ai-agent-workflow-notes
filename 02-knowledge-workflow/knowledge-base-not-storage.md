# 知识库不是资料堆积

> 版本：v1 基础稿  
> 关键词：Knowledge Workflow / 上下文 / Obsidian / AI 可读性

## 一句话

知识库不是把资料存起来，而是把资料整理成“人能复盘，AI 能读取，下一次能复用”的上下文系统。

如果说文件夹是仓库，知识库更像厨房：不是把食材全塞进去，而是要知道什么东西放在哪里，什么时候能拿出来做菜。

## 为什么资料堆积没有用

很多人做知识库，会很快陷入一个问题：收藏越来越多，但真正用的时候找不到、想不起、接不上。

这通常不是工具问题，而是结构问题。

常见的资料堆积有三个特征：

- 只保存，没有加工
- 只分类，没有场景
- 只记录，没有下一步动作

这样的知识库，对 AI 也不友好。因为 AI 不是缺资料，而是缺清楚的上下文。

## 我理解的知识工作流

一个能被 AI 使用的知识库，至少要回答四个问题：

| 问题 | 作用 |
| --- | --- |
| 这是什么 | 让资料有基本身份 |
| 为什么重要 | 让资料有判断价值 |
| 可以用在哪 | 让资料连接到场景 |
| 下一步怎么用 | 让资料变成行动入口 |

所以我更关心的不是“存了多少资料”，而是“这些资料能不能帮我更快做出一个判断、产出一份内容、完成一个工作流”。

## 一个具体例子

比如我看到一篇关于 AI Agent 的文章。

资料堆积的做法是：收藏链接，放进 AI 文件夹。

知识工作流的做法是：

1. 摘出它讲的核心概念；
2. 标注它适合哪个场景，比如求职、交付、内容、工具；
3. 写一句我自己的理解；
4. 判断它能不能变成 prompt、流程卡、网页内容或项目说明。

这样，它就不是一个“以后可能会看”的资料，而是进入了我的工作系统。

## 我的最小整理格式

我现在会用这个轻量模板处理资料：

```md
# 标题

## 这是什么
一句话说明。

## 对我有什么用
连接到我的真实场景。

## 可复用内容
可以变成 prompt / 流程 / 文章 / 项目说明的部分。

## 下一步
一个具体动作。
```

这个格式不复杂，但足够让资料从“存档”进入“可使用”。

## 当前结论

知识库的重点不是完整，而是可调用。

对我来说，好的知识库不是第二大脑的摆设，而是 AI 工作流的燃料站：资料被整理好，Agent 才能真的帮我做事。
